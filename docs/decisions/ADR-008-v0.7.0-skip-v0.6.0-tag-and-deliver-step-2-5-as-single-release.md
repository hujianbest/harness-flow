# ADR-008: 跳过 v0.6.0 tag，把 ADR-007 D3 Step 2-5 一次性合并为 v0.7.0 完整发布

- 状态：起草中（2026-05-10 锁定，待 features/002-leaf-skill-decoupling closeout 翻 accepted）
- 决策人：用户（架构师角色）
- 工程团队：HF（按 `docs/principles/soul.md` 协作契约执行）
- 关联 feature: `features/002-leaf-skill-decoupling/`（v0.7.0 主候选）
- 关联文档：
  - `docs/decisions/ADR-007-orchestrator-extraction-and-skill-decoupling.md`（D1 + D3 锁定 6 步落地路径；本 ADR 是 D3 Step 2-5 的执行级范围决策）
  - `features/001-orchestrator-extraction/`（v0.6.0 中间产物，含 closeout.md；本 ADR 锁定该 feature 已交付的 Step 1 不在 v0.7.0 中重复发布）
  - `CHANGELOG.md`（[0.6.0] 段已写但 v0.6.0 不打 tag；v0.7.0 段整合两段交付内容）

## 背景

ADR-007（v0.6.0 minor scope）锁定了 HF 三层架构 invariant + 6 步落地路径：

- Step 1（v0.6.0 范围）: 建 `agents/hf-orchestrator.md` + 9 references 物理迁移 + 3 宿主 always-on stub + 旧 skill 转 deprecated alias
- Step 2-5（推迟到 v0.7+）: leaf skill `Next Action` 字段降级 / Hard Gate 分级 / `[Workflow]` 类 Gate 物理上提 / 跨 hf-* 引用清理
- Step 6（推迟到 v0.8+）: 物理删除 `using-hf-workflow` / `hf-workflow-router` deprecated stub

**v0.6.0 已合并到 main**（PR #45 / #46 / #47），但**未打 tag**。原因：

1. **架构 narrative 与实际交付不对齐**。ADR-007 D1 声明"Layer 1/Layer 2 互不引用"是架构 invariant，但 v0.6.0 范围严格限定 Step 1，**24 个 leaf skill 一行没改**——它们仍然写 `Next Action Or Recommended Skill: hf-X`、宣告"路由权属于 hf-workflow-router"、Hard Gates 中"必须存在已批准 tasks.md"等耦合都还在。
2. **原始目标（standalone-usable skills）未达成**。用户用此项目最初提出的痛点（leaf skill 不能单独调用、不满足 Anthropic Agent Skills 原始定位）在 v0.6.0 范围内**没有改善**——leaf 不解耦、单独调用照样被 hard gate 拦住。
3. **ADR-007 D1 Amendment 已经入档**承认 v0.6.0 = "architectural commitment"（架构承诺）状态而非 runtime enforcement，但用户决策是**不接受半成品发布**——一次性兑现 Step 2-5 后再发布。

ADR-001 D1 立场"P-Honest, narrow but hard"在 v0.6.0 narrative 上有滑坡风险（如果 release notes 写"standalone-usable skills"会被立刻打脸）。本 ADR 以**修正路径**重新对齐 narrative 与实际交付。

## 决策

### Decision 1 — 跳过 v0.6.0 tag；不在 GitHub Releases 创建 v0.6.0 pre-release

`v0.5.1` 是上一个 tag。下一个 tag 直接是 **`v0.7.0`**，包含：

- v0.6.0 已合并到 main 的 commits（PR #45 + #46 + #47；ADR-007 + orchestrator 文件 + 9 references 迁移 + 3 宿主 always-on stub + deprecated alias 等）
- v0.7.0 新增的 Step 2-5 实施内容（24 leaf skill 解耦修改 + orchestrator 升级到纯 artifact 驱动）

不在 git history 上 force-push 删除 v0.6.0 commits——它们是真实工作记录，保留作为 v0.7.0 的实施基础。仅"不打 tag、不在 GitHub Releases 公开宣传"。

### Decision 2 — v0.7.0 范围 = ADR-007 D3 Step 2 + Step 3 + Step 4 + Step 5（**不**含 Step 6）

明确 in-scope:

- **Step 2**: 24 个 hf-* leaf skill 的 `Next Action Or Recommended Skill` 字段从必填降为可选；orchestrator 同步支持"无 Next Action 字段时基于 on-disk artifacts 自行决定"
- **Step 3**: leaf skill 的 Hard Gates 显式分类标注 `[SOP]` / `[Workflow]`；`[SOP]` 类是 skill 单独调用时仍生效的纪律（如 TDD Two Hats / SUT Form）；`[Workflow]` 类是只在 HF workflow 编排下成立的依赖（如 "已批准 tasks.md"）
- **Step 4**: `[Workflow]` 类 Hard Gate 物理从 leaf 移到 orchestrator persona；orchestrator 新增 pre-check 阶段在 invoke leaf 前校验
- **Step 5**: 删除 leaf skill 里所有对其它 hf-* 的硬引用（包括 SKILL.md body 的"由 hf-workflow-router 决定"、"由 X handoff 给 Y"、reviewer return contract 中的下游 skill ID 硬指等）；改为软提及（仅在 `## See Also` 段出现）或上提到 orchestrator side

明确 out-of-scope（推迟到 v0.8+ 或更后续）:

- **Step 6**: 物理删除 `skills/using-hf-workflow/` 与 `skills/hf-workflow-router/` 及 9 个 references stub。继续保留 v0.6.0 状态（using-hf-workflow 是 plugin-install 加载通道；hf-workflow-router 仍是 deprecated alias）。理由：留 v0.7.x 一个完整 minor 周期给外部教程链接 grace period
- **5 项原 deferred ops/release skills**（hf-shipping-and-launch / hf-ci-cd-and-automation / hf-security-hardening / hf-performance-gate / hf-deprecation-and-migration / hf-debugging-and-error-recovery）：维持 ADR-005 D7 + ADR-007 D6 立场，继续 deferred
- **3 个 specialist personas**（hf-staff-reviewer / hf-qa-engineer / hf-security-auditor）：维持 ADR-007 D7 立场，继续 deferred
- **4 家 deferred 客户端**（Gemini CLI / Windsurf / GitHub Copilot / Kiro）：维持 ADR-003 D2 立场

### Decision 3 — v0.7.0 不重新 spec/design；继承 ADR-007 + 走 hf-test-driven-dev 直接 implementation

理由：架构决策（D1 三层 invariant + D3 6 步路径 + 各 step 语义）已在 ADR-007 完整锁定 + reviewer 通过；v0.7.0 是 D3 Step 2-5 的**执行级落地**，不是新架构决策。重做 spec/design 是 ceremony 浪费。

具体安排：
- 创建 `features/002-leaf-skill-decoupling/`，含 README.md + progress.md + tasks.md
- 不创建独立 spec.md（继承 ADR-007 D1 + spec FR-001..007）
- 不创建独立 design.md（继承 ADR-007 D3 Step 2-5 + features/001 design D-X 决策）
- tasks.md 直接从 ADR-007 D3 Step 2-5 派生
- 派发 hf-tasks-review 评审 tasks.md
- approval 后进 hf-test-driven-dev 实施
- 实施后走完整 review chain（hf-test-review / hf-code-review / hf-traceability-review / hf-regression-gate / hf-completion-gate）+ hf-finalize + hf-release

### Decision 4 — Tiered 实施策略：Tier 1 高频 leaf 优先 + Tier 2 其余

**23 个**活跃 hf-* leaf（12 doer + 11 reviewer/gate per ADR-007 D1）不一刀切并行，分两 Tier 顺序执行以控制风险。`skills/using-hf-workflow/` 与 `skills/hf-workflow-router/` 是 v0.6.0 deprecated alias，**不**计入 leaf：

**Tier 1**（10 leaf；高频 / dogfood 用得到 / release-blocking 验证依赖）:

| Doer (4) | Reviewer (6) |
|---|---|
| hf-specify | hf-spec-review |
| hf-design | hf-design-review |
| hf-tasks | hf-tasks-review |
| hf-test-driven-dev | hf-test-review |
|  | hf-code-review |
|  | hf-traceability-review |

**Tier 2**（13 leaf；剩余 8 doer + 2 reviewer + 3 gate）:

| Doer (8) | Reviewer (2) | Gate (3) |
|---|---|---|
| hf-product-discovery | hf-discovery-review | hf-regression-gate |
| hf-experiment | hf-ui-review | hf-completion-gate |
| hf-hotfix |  | hf-doc-freshness-gate |
| hf-increment |  |  |
| hf-finalize |  |  |
| hf-ui-design |  |  |
| hf-browser-testing |  |  |
| hf-release |  |  |

10 + 13 = **23 leaf 全覆盖**。每 Tier 完成后做 sub-gate 检查（grep 验证 4 步全做 + Hard Gate 标签 + Next Action 已可选 + 跨 hf-* 引用为 0），通过后进入下一 Tier 或 release pack。

### Decision 5 — HYP-005 升级为 v0.7.0 release-blocking 假设

ADR-007 D5 已锁定 HYP-002 + HYP-003 为 v0.6.0 release-blocking。v0.7.0 新增 **HYP-005**（`features/001-orchestrator-extraction/spec.md` § 4 已定义；v0.6.0 范围内标 non-blocking）升级为 v0.7.0 release-blocking：

> HYP-005: leaf skill 剥离 `Next Action Or Recommended Skill` 字段后，orchestrator 仍能基于 on-disk artifacts（progress.md / reviews/ / verification/ / spec/design/tasks artifacts）决定下一步。

验证手段：walking-skeleton 端到端运行（**升级**到运行时等价证明，不再仅 self-diff 静态等价；HYP-002 同时升级）。具体：

1. 跑一遍完整 v0.7.0 walking-skeleton，从 spec → finalize；不依赖 leaf 的 `Next Action` 字段
2. 对照 v0.5.1 walking-skeleton baseline 跑同一条端到端轨迹
3. 产出物 schema-by-schema 等价（容许差异白名单不变）
4. 必要时在新 walking-skeleton demo（如 examples/v0.7.0-leaf-decoupling-demo/）做 fresh evidence

任一失败 → v0.7.0 release rollback；不打 tag。

### Decision 6 — CHANGELOG 整合为单 [0.7.0] section；删除 [0.6.0] section

由于 v0.6.0 不打 tag、不公开发布，CHANGELOG 上不应保留独立 [0.6.0] 段（外部读者会困惑"v0.6.0 是不是被悄悄发了"）。

实施：把 [0.6.0] 段内容**整合**进新的 [0.7.0] 段，作为该 release 的"Foundation work (originally drafted as v0.6.0; now consolidated)"子段。明确说明 v0.7.0 包含两批工作：

1. v0.6.0 阶段已合并的架构 foundation（Step 1）
2. v0.7.0 阶段新增的 Step 2-5 实施

[0.7.0] 段标题写明这是首个**真正交付 standalone-usable skills** 的 release。

### Decision 7 — Release narrative 校准（README / setup docs / 长期资产）

按 ADR-001 D1 "P-Honest, narrow but hard" 立场，v0.7.0 的对外 narrative 必须诚实：

- README Scope Note 重写为 v0.7.0；明确说"v0.6.0 was prepared but not tagged—superseded by v0.7.0"
- setup docs 重写描述（Step 1 + Step 2-5 全部交付）
- ADR-007 D1 Amendment 在 v0.7.0 范围内**仍然适用**——plugin-install 加载通道仍走 `skills/using-hf-workflow/SKILL.md`（继续 v0.6.0 决定的物理通道分工）
- HF dogfood 故事："这是 HF 自己用 HF 流程修复 HF narrative-vs-reality gap 的实例"——可作为 release notes 的诚实呈现，不掩盖

## 不做（明确边界）

承接 ADR-001 D1 / ADR-005 D7 / ADR-007 D6 / D7 立场延续：

- 不引入新 hf-* skill；保持 24 个不变
- 不引入新 slash 命令；保持 7 条不变
- 不修改 closeout pack schema / reviewer return verdict 词表 / `hf-release` 行为 / `audit-skill-anatomy.py` / `hf-finalize` step 6A
- 不引入 specialist personas；`agents/` 目录仍只有 `hf-orchestrator.md`
- 不删除 deprecated alias（D3 Step 6 仍 deferred 到 v0.8+）
- 不做部署 / 监控 / 回滚 / 安全加固 / 性能 gate / 依赖弃用迁移 / 调试与错误恢复——延续历版立场

## 风险与开放问题

### 接受的工程风险

- **R1**: 24 leaf 批量修改可能引入回归（修改面 ~1200 行）。Mitigation: Tiered 实施 + 每 Tier sub-gate + walking-skeleton 端到端 release-blocking 验证
- **R2**: Orchestrator 升级到纯 artifact 驱动后，决策可能在 ambiguous 状态下产生不同路由。Mitigation: 保留 v0.6.0 兼容期 fallback（仍可消费 leaf `Next Action` 作为辅助 hint）；only when leaf 不再写该字段时才纯 artifact 驱动
- **R3**: HYP-005 验证失败会导致 v0.7.0 不可发布。Mitigation: D3 Step 2-5 解耦时 orchestrator 同步升级；任一 Tier 验证失败立即 stop + diagnose

### 留给后续节点的开放问题

- v0.7.x 是否需要做小颗粒度 patch release（如 Step 6 物理删除单独 v0.8.0；hf-shipping-and-launch 引入单独 v0.8.0+ minor）？由后续 ADR 单独决定
- skill anatomy v3（agents anatomy audit）是否需要引入？ADR-007 D2 已留待后续 ADR 决定，本 ADR 不开启
