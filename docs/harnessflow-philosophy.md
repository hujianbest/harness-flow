# HarnessFlow 核心理念

> 本文是 HarnessFlow 的**核心理念纲领（philosophy）**：它陈述 HarnessFlow 是什么、为什么存在、以及它如何看待「高质量代码」。
>
> 它是 HarnessFlow 一切设计的**北极星**。任何具体的技能、命令、agent 与流程，都应当回溯并服从于本文；当实现与理念冲突时，以理念为准。
>
> 本文与具体工具、编程语言、领域无关，意在长期存活。实现细节不属于本文。
>
> 实现架构见 [`harnessflow-core-architecture.md`](harnessflow-core-architecture.md)。

---

## 0. 一句话主张

> **HarnessFlow 是一套跨平台的 AI 软件开发工作流框架，目标是在高质量要求的软件开发中，指导 AI 模型稳定地产出高质量代码。**

它把「产出高质量代码」这件事拆成**由外到内**的三层质量模型——**SDD（规范驱动）→ TDD（测试驱动）→ 整洁代码与软件设计**——并以 **human-on-the-loop**（人在环上审查、AI 在环内干活）为根本协作姿态。

---

## 1. 为什么需要 HarnessFlow

AI 编码的默认路径是：**从含糊的一句话需求，直接跳到代码。** 这条捷径会稳定地产出三类失败，而这三类失败正是 HarnessFlow 三层模型各自要对抗的靶子：

| 失败模式 | 表现 | 后果 |
|---|---|---|
| **做错了事** | 需求含糊，模型只能靠猜 | 做出来的不是你真正想要的东西 |
| **做得不对** | 代码没有被验证，留一堆 BUG | 验证负担被甩回给人，「看起来能跑」≠「真的对」 |
| **做得不好** | 代码能跑，但写得烂 | 难维护、难演进、难审查，技术债持续累积 |

这三类失败可以更精确地表述为：**意图不正确 / 功能不正确 / 内在质量不足**。它们不是同一类问题的轻重之分，而是三道性质不同的关口——意图错了，写得多漂亮都是白费；功能没被证明，再多的「看起来能跑」也只是把验证债务转嫁给人；内在质量差，代码即便此刻正确，也活不过下一次改动。HarnessFlow 的存在，就是把这条草率的捷径替换成一条**更窄、更难、但能产出可信结果**的路径：先讲清楚要做什么，再证明它做对了，最后把它写好。

HarnessFlow 把这条路径固化为一组可复用、可审计的技能与工件约定。它的前提是承认：AI 模型在单步推理上很强，但在长链路、多约束、需要持续纪律的工程任务上会稳定滑坡——会跳过规范、省略测试、堆砌「能跑就行」的代码。框架的存在，正是用流程把模型强制约束在高质量路径上，而不是寄望于模型每次都自觉。

## 2. 三层质量模型

HarnessFlow 用三层来组织「高质量」。理解这三层，需要先抓住三个要点：

- **外部质量 vs 内在质量**：前两层（SDD、TDD）保证代码的**外部质量**——它做的是不是对的事、功能上对不对；第三层保证代码的**内在质量**——它本身写得好不好。
- **由外到内、递进且带反馈环**：三层以「由外到内」为主线逐步收紧（先保证做对的事，再保证做对，最后保证做好）；但这不是僵硬的瀑布——高层发现的问题允许**反哺**低层（例如写测试时暴露了规范的漏洞，就回头补规范）。
- **三层不是三个产物，而是看待同一份代码的三个维度**：三层最终都收敛到同一份代码上，只是关注代码的不同侧面。它们不被严格隔离，而是共同体现在代码里。

### 三层质量模型表

| 层 | 回答的问题 | 失败模式（无此层时） | 承载技能 |
|---|---|---|---|
| 第一层 SDD | 做的是不是对的事？ | 需求含糊 → 模型靠猜 → 做错事 | hf-specify |
| 第二层 TDD | 功能被证明正确了吗？ | 代码未验证 → 一堆 bug 给人 | hf-tdd |
| 第三层 Clean Code | 代码本身写得好吗？ | 能跑但烂 → 难维护/审查/演进 | hf-clean-code + *-coding-standards + 领域技能 |

### 第一层 · SDD（规范驱动开发）— 外部质量 · 意图正确

- **关注代码的哪个维度**：代码是否做了**对的事**（是否符合需求意图）。
- **目的**：向模型**清晰地**描述需求——具体要做什么、约束是什么、什么算完成，而不是一句含糊不清的描述。规范是模型与人之间的第一份契约。
- **对抗的失败模式**：需求含糊 → 模型靠猜 → 做错了事。
- **人类把关点**：人确认并审查规范，确保「要做的事」本身是对的。

### 第二层 · TDD（测试驱动开发）— 外部质量 · 功能正确

- **关注代码的哪个维度**：代码是否被**验证为功能正确**。
- **目的**：保证模型生成的代码从功能上看是能正确运行的，**不会留一堆 BUG 等着人工去验证**。测试把「正确」从主观判断变成可执行、可复现的事实。
- **对抗的失败模式**：代码未经验证 → 留一堆 BUG → 验证负担甩给人。
- **人类把关点**：人确认测试真正表达了正确的行为期望，而非只是凑过的断言。

### 第三层 · 整洁代码与软件设计 — 内在质量

- **关注代码的哪个维度**：代码本身**写得好不好**。
- **目的**：编写**整洁的代码**并做出**良好的软件设计**——简洁、可维护、可靠、可测试、高效，整体上**高内聚、低耦合**。
- **因果关系（重要）**：HarnessFlow 在这一层真正追求的是**代码整洁与良好设计本身**。可长期演进、以及易于人审查（让 human-on-the-loop 得以低成本运转），是整洁与良好设计**自然带来的回报**，而**不是**与「整洁」并列、需要单独去追求的目标。换句话说：**把代码写整洁了，可演进性与可审查性自会随之而来。**
- **对抗的失败模式**：能跑但烂 → 难维护、难演进、难审查。
- **人类把关点**：人审查设计与可读性，判断这份代码是否值得长期持有。

### 小结

> 前两层（外部质量）保证代码是**对的、能用的**；第三层（内在质量）保证代码是**好的、能长期活下去、人审得动的**。三者由外到内层层递进，又共同体现在同一份代码的不同维度上。

## 3. 主流程闭环

三层质量模型不是抽象的检查项，而是被钉死在一条主流程闭环上。每个阶段产出工件、走过一个独立评审（R 节点），再进入下一阶段。主流程逐字如下：

```text
specify → R1 → design → R2 → tdd → R3 → ship
```

缺陷不走完整主流程，而是从修复点接回正轨。缺陷旁路逐字如下：

```text
fix → tdd → R3 → ship
```

这条闭环的关键纪律是 TDD 内部的三色循环：**GREEN 改行为、REFACTOR 改表达，两者不混在一个 diff。** RED 先证明测试有效（让它失败），GREEN 用最小改动让它转绿，REFACTOR 在绿色保护下改写代码的结构与命名。把行为改动与结构改动塞进同一个 diff，是审查无法判读、回归无法定位的常见根因——一旦两件事耦合，失败时你不知道是「新行为错了」还是「重构破坏了旧行为」。这条纪律看似拖慢节奏，实际上是把可回滚、可审查的下限兜住了。

## 4. 协作姿态：Human on the loop

HarnessFlow 刻意选择 **human-on-the-loop**，并与 human-in-the-loop 区分开：

| 姿态 | 谁在干活 | 人的角色 |
|---|---|---|
| **human-in-the-loop** | 人和 AI 一起下场干活 | 参与到具体执行环节中 |
| **human-on-the-loop**（HarnessFlow 采用） | **具体的活由 AI 干** | **人站在「环上」做审查与把关** |

这一姿态贯穿三层，并有两个直接推论：

1. **每一层的产出，本质上都是「AI 生产、人审查」的交接物。** 因此每一层都必须存在明确的人类把关点（审规范、审测试与行为、审设计与可读性）。
2. **第三层的整洁是这一姿态能否成立的关键。** 人之所以能持续、低成本地站在环上掌控 AI 的产出，而不是被淹没在不可审的代码里，正是因为代码本身整洁、设计良好、易于审查。

### 证据优于记忆

human-on-the-loop 要成立，审查的对象就不能是模型的「记忆」或聊天里的口头承诺，而是落到磁盘上的**工件**：spec、测试证据、评审记录、设计文档、追溯矩阵。一句「我已经测试过了」不构成证据；plan.md 里 RED→GREEN 的证据行才构成。新会话续作时不依赖聊天历史，只读工件即可重建进度——这是 HarnessFlow 的 artifact-first 恢复原则。

---

## 5. 三条硬规则

角色分离是独立评审得以成立的前提，固化成三条硬规则：

1. **作者不自审**：实现者不评审自己的产出，评审由独立的 hf-reviewer 执行。
2. **评审者不动手修**：评审只产出 findings 与 verdict，修改回到作者手里执行。让评审者边评边改，等于把「独立判断」和「亲手实现」的利益冲突合并到了同一个人身上。
3. **人最终把关**：规格确认、设计确认、评审 verdict 闭环、DoD 核验后的关闭，都保留人的把关点。HarnessFlow 不替团队拍板业务方向、优先级、验收阈值、架构边界。

这三条规则共同把「AI 生成、人审查」变成可重复、可问责的交接，而不是一句口号。

## 6. 派生设计原则

以下原则从三层质量模型与 human-on-the-loop 逻辑推导而来。它们**与任何具体实现无关，但约束所有实现**——后续的技能与命令应当对照这些原则自检：

1. **规范先行于代码**：没有清晰、可审查的规范，就不进入实现。
2. **测试是验收，而非事后补丁**：用测试定义「正确」，先让其失败、再让其通过，把正确性变成可复现的事实。
3. **整洁优先，可审查性随之而来**：把内在质量当作一等目标去追求；不要把「可审查」当成可以单独凑出来的指标。
4. **每一层都要有明确的人类把关点**：AI 生产、人审查；任何一层都不能跳过人的把关。
5. **由外到内，但允许反馈回溯**：以递进为主线，同时承认高层洞察会反哺低层，并为这种回溯留出路径。
6. **通用内核与领域特化分离**：核心理念（三层模型 + human-on-the-loop）保持通用；特定领域的质量维度作为可按场景披露的扩展，而非内核的一部分。

---

## 7. 一句话目标

把以上全部收敛成一句可背诵的目标：

> **在 SDD 范式下写 Clean Code，而不是仅仅能运行的代码。**

这句话同时框定了 HarnessFlow 的下限与上限：下限是「必须先有规范、再用测试证明」，上限是「证明对了之后，还要把代码本身写好」。仅能运行的代码只是起点；只有当它既被规格驱动、又被测试证明、还符合整洁标准时，才达到 HarnessFlow 所定义的「高质量」。

---

## 8. 适用范围与源起

- **适用范围**：HarnessFlow 的核心理念面向**任何高质量要求的软件开发**，与编程语言、技术栈、领域无关，并设计为跨 Claude Code、Cursor、OpenCode 等运行时复用。
- **源起**：HarnessFlow 的三层质量模型与协作姿态继承自 DevFlow 的工程实践，并针对多平台分发与可扩展的技能契约做了重构。与具体语言、平台或行业强相关的质量维度（内存、并发、实时性、资源、ABI、行业合规等）作为独立的语言/领域扩展技能，不属于本文所定义的核心理念。
