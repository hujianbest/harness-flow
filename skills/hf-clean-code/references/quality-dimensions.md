# 五维质量判据

> 配套 `hf-clean-code`。用于实现期 `clean_code_check`、R3 代码评审、纯重构计划和 ship 前约束审计。五维不是五份清单，而是同一段代码是否值得长期持有的五个读法。

## 使用方式

1. 先确认范围：只检查当前任务/当前 diff 引入或触碰的问题；旧问题记录为债务，不借机重写无关区域。
2. 先判影响，再判严重级：会破坏正确性、状态、资源或可审查性的异味，不能降级成风格偏好。
3. 先找最小动作：能在当前任务内 Extract Function / Rename / Guard Clause 解决，就留在 `hf-tdd`；需要改接口契约、模块边界、错误模型或变化轴，回 `hf-design`。
4. 所有重构都在全绿上进行；没有测试基线时先补足行为保护。

## 1. 简洁

**目标**：当前需求需要的最少结构，能直接读出主路径。

| 坏信号 | 为什么重要 | 动作 |
|---|---|---|
| 为"以后可能"添加插件点、注册表、单实现接口 | 假想变化轴制造长期维护成本 | 删除投机抽象；保持具体实现 |
| 函数同时含业务流程和底层细节 | 读者无法在一个抽象层理解意图 | Extract Function，把细节下沉 |
| flag 参数切换两套行为 | 一个名字掩盖两个职责 | Split Flag Parameter |
| 大量配置项只是暴露内部实现 | API 表面扩大，未来难收回 | 回设计确认真实扩展点 |

**阻塞**：设计未批准的新抽象、单实现接口、插件点进入生产路径，通常 important；若改变边界或契约，回 `hf-design`。

## 2. 可靠

**目标**：错误路径、资源释放和失败状态与主路径一样清楚。

| 坏信号 | 为什么重要 | 动作 |
|---|---|---|
| 可失败调用未检查 | 失败变成隐式状态污染 | 检查返回值/异常；忽略必须有理由 |
| catch 后只打日志继续 | 调用方以为成功，系统状态可能已错 | 恢复、降级、上报或返回错误 |
| 多资源获取有多个提前返回 | 新增资源时容易漏释放 | 用语言惯用的集中清理出口或确定性资源管理 |
| 错误信息无上下文 | 线上问题不可定位 | 带对象、索引、实际值、期望值 |

**阻塞**：吞错误、资源泄漏、失败状态违反 design 契约按 critical；修复若要求重定错误模型，回 `hf-design`。

## 3. 可维护

**目标**：名字、职责和重复关系让下一次修改位置显而易见。

| 坏信号 | 为什么重要 | 动作 |
|---|---|---|
| 名字撒谎 | 调用方会基于错误语义使用它 | 改名或拆函数；注释不能替代 |
| 一个文件/类处理多个变化理由 | 修改一个需求牵动无关行为 | Move Function / Extract Module；大边界回设计 |
| 第三次重复同一知识 | 规则会分叉，bug 修一漏二 | 提取常量、函数或参数对象 |
| 大接口暴露内部结构 | 调用方依赖实现细节 | 拆小 API / opaque handle / 隐藏内部字段 |

**阻塞**：命名撒谎、上帝对象、大接口污染、未登记长函数/深嵌套通常 important；影响模块边界时回 `hf-design`。

## 4. 可测试

**目标**：生产代码通过公共行为可验证；测试代码本身可读、稳定、可信。

| 坏信号 | 为什么重要 | 动作 |
|---|---|---|
| 为测试暴露 public test-only 方法 | 生产 API 被测试需要污染 | 通过公共行为观察；必要时回设计 |
| mock 内部纯逻辑 | 测试锁住实现方式，不验证行为 | mock/fake 只用于真实边界 |
| 测试依赖真实时间、随机数、顺序 | flaky 或不可复现 | 注入 clock/random；每测例独立 setup |
| helper 抽象过深 | 读者看不懂 Given/When/Then | 保留必要重复，让行为在测试体中可见 |

**阻塞**：test-only 生产后门按 critical；测试无法失败、共享状态、mock 边界错误通常 important，并回 `hf-tdd` 的测试质量修复。

## 5. 高性能

**目标**：性能与资源成本匹配真实使用场景，不用无证据的复杂技巧换取表面优化。

| 坏信号 | 为什么重要 | 动作 |
|---|---|---|
| 热路径 N+1 查询/调用 | 数据规模增长时成本爆炸 | 批量获取、缓存边界、调整数据结构 |
| 无界读取/循环/递归 | 输入变大导致延迟、内存或栈风险 | 加分页、上限、流式处理或迭代 |
| 重复解析、分配、计算在循环内 | 浪费资源并制造抖动 | 提前计算、复用结构、缓存纯结果 |
| 为性能写晦涩技巧但无证据 | 牺牲可维护性，未来难改 | 恢复清晰写法或附基准/约束说明 |

**阻塞**：热路径明显 N+1、无界读取、资源泄漏按 important 到 critical；没有测量或边界证据的微优化不应阻塞合并，但可登记观察。

## 升级速查

| 发现 | 默认去向 |
|---|---|
| 局部命名、函数拆分、卫语句、常量命名 | `hf-tdd` REFACTOR |
| 测试断言弱、fixture 混乱、mock 边界错误 | `hf-tdd` RED/GREEN 或测试重构 |
| 错误路径、资源路径 bug | `hf-tdd` 修复；必要时先写失败测试 |
| 接口契约、错误模型、模块边界、变化轴错误 | `hf-design` |
| 规格或验收标准导致代码必须猜 | `hf-specify` |
| 超出任务范围但值得做 | plan.md 债务节 / 新工作项 |
