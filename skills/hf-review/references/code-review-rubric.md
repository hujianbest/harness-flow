# 代码评审 Rubric

> 评审门禁：R3（与测试评审同轮）。评审对象：实现 diff。上游输入：`design.md`、`spec.md`、适用的 `<language>-coding-standards` 与领域技能。作者侧标准见 `hf-clean-code`。
>
> 核心怀疑：**这段代码哪里在对读者撒谎？错误路径和资源路径先读。**

## 检查项

### 正确性与一致性（不过 = critical）

- [ ] 实现行为与 `design.md` 一致；偏离处有显式说明且工件已更新（代码与设计默默不一致 = critical）
- [ ] `modify`/`remove` 的基线语义正确落地，无未批准的既有行为破坏
- [ ] 边界条件无 off-by-one / 截断 / 溢出遗漏；无逻辑上不可达却被依赖的分支
- [ ] `traceability.md` 每条受影响需求行的代码文件/函数、测试代码、验证证据列非空，或有明确 `N/A` 理由；抽查 2-3 行与 diff 实际内容一致（漂移 = critical）

### 错误路径与资源（优先细读，不过 = critical）

- [ ] 每个可失败调用被检查，或显式注明忽略理由
- [ ] 每条错误路径：已获取的资源被回收（成对获取/释放；C 集中清理出口 / C++ RAII / Python with / Java try-with-resources）
- [ ] 失败后状态符合契约（回滚/保留/明确的中间态）；无吞错误（捕获后既不处理也不上报）
- [ ] 错误信息带定位上下文

### 并发与生命周期（适用时）

- [ ] 共享数据访问受保护；锁顺序一致；临界区精简
- [ ] 中断上下文限制遵守（无阻塞调用/动态分配；具体规则见适用的领域开发技能）
- [ ] 跨边界指针的存活期符合设计的所有权约定；无悬垂风险

### 整洁标准（按 hf-clean-code）

- [ ] 命名不撒谎：函数名说出全部行为；布尔肯定语义；量有单位
- [ ] 函数单一职责、单一抽象层；无未登记的 >50 行 / >4 参数 / >3 层嵌套
- [ ] 主路径低缩进、卫语句处理异常；无裸魔法数
- [ ] 注释解释 why 不复述 what；无注释掉的代码、无死代码、无 test-only 后门
- [ ] 重复代码：第三次出现的相同知识已提取；未提取「长得像但知识不同」的代码
- [ ] SOLID 编码落点：无上帝对象/大而全接口；无破坏契约的伪多态；无高层流程直接依赖底层细节（除非设计说明允许）
- [ ] 新增抽象有真实边界或真实变化轴；无只为 mock 或「以后可能」创建的单实现接口
- [ ] 测试代码同样可读、独立、可重复；无过度 helper 抽象、共享可变状态或 mock 内部纯逻辑
- [ ] 热路径无明显 N+1、无界读取、重复重计算或资源生命周期不清；无证据微优化不阻塞但可登记

以下整洁问题不是风格偏好，按 blocking/critical 处理：命名撒谎导致调用方误解行为；吞错误或错误路径状态不符合契约；未登记的大函数/深嵌套掩盖多职责或错误路径；spec/design 未批准的新抽象、单实现接口、插件点；test-only 后门进入生产代码；热路径中有证据的 N+1/无界资源风险。它们会让代码「能跑但不可维护」，必须修复或回 `hf-design`。

### 范围与 diff 卫生

- [ ] diff 只含本任务范围；行为变更与重构/格式化未混在一个提交
- [ ] 无顺手修改的无关文件；路过问题已登记而非默默修掉
- [ ] 构建无新增警告；静态分析新增项已修复或带理由抑制

### 语言与领域规则

- [ ] 适用语言的 `<language>-coding-standards` 规则已过；按 diff 涉及的语言加载对应技能
- [ ] 命中 description 的领域开发技能已加载；适用项有实现或显式 N/A，未命中的领域有不适用理由

## Verdict 指引

- 正确性/错误路径/一致性 critical → `需修改`
- blocking/critical 整洁问题 → `需修改`；若反映模块边界或抽象方向错误，verdict `重新设计`
- 结构性问题超出任务边界（模块划分错误、接口契约缺陷）→ `重新设计` 指向 `hf-design`
- 纯格式、局部命名偏好等 minor 不阻塞，列出由人决定取舍；不要把上面列出的 blocking 问题降级成 minor
