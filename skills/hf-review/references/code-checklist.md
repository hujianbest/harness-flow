# 代码评审 Checklist

对照实现代码、测试、已批准的 `design.md` 与 `spec.md` 逐条检查。加载的 `ext-*` 扩展若声明了实现阶段的追加检查项,一并检查。

## 需求追溯

- [ ] 任务清单全部勾选完成,每个任务有本会话可复现的测试证据(命令 + 结果)
- [ ] 规格的每条 FR 有对应的通过测试;每条 NFR 有验证证据
- [ ] 没有超出规格与设计范围的改动;有则必须已向用户说明

## 测试质量

- [ ] 测试验证的是行为(输入→可观察输出),不是实现细节(私有函数、内部调用次序)
- [ ] 覆盖边界值与失败路径,不只有 happy path
- [ ] 无造假测试:断言总为真、断言被注释、为通过而放宽断言、测试里复制实现逻辑
- [ ] mock 只用于真正的外部边界(网络、时钟、文件系统),没有 mock 被测对象本身
- [ ] 全量测试套件在当前状态下运行通过(不只是新增测试)

## 设计符合度

- [ ] 实现遵循设计的模块划分、依赖方向与接口契约;偏离处有记录与理由
- [ ] 错误处理与设计声明的失败模式一致,没有静默吞错

## 代码质量

- [ ] 命名清晰;无复制粘贴式重复;无死代码、被注释掉的代码、调试残留
- [ ] 没有引入设计未声明的新抽象层
- [ ] 遵循项目既有代码风格与惯例
