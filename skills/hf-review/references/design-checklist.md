# 设计评审 Checklist

对照 `design.md` 与已批准的 `spec.md` 逐条检查。加载的 `ext-*` 扩展若声明了设计阶段的追加检查项,一并检查。

## 需求覆盖

- [ ] 规格的每条 FR/NFR 都能在设计中找到落点;任务清单"覆盖需求"列穷尽全部 FR/NFR
- [ ] 设计没有引入规格之外的新需求或新能力
- [ ] 每条 NFR 有明确的满足机制与验证方式

## 决策质量

- [ ] 每个关键决策列出 ≥2 个方案并有可检验的取舍分析,而非"直接选定"
- [ ] 无过度设计:没有为未确认的未来需求引入的抽象层、接口、配置项
- [ ] 与现有代码的关系明确:复用什么、新增什么;偏离既有模式的地方有理由

## 可实现性

- [ ] 接口与数据契约具体到实现阶段不需要再发明(签名、结构、错误模型)
- [ ] 关键路径的失败模式有对策
- [ ] 测试策略可执行:分层明确、有运行命令、指明必须覆盖的边界

## 任务清单

- [ ] 每个任务一次 TDD 循环内可完成,有可判定的完成判据
- [ ] 任务按依赖排序,首个任务打通最薄端到端路径
- [ ] 没有"研究一下""优化性能"这类无判据任务
