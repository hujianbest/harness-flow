# 设计评审 Rubric

> 评审门禁：R2。评审对象：工件根下 `features/<id>-<slug>/design.md`，以及（影响组件边界时）`component-design-draft.md`（或团队覆盖路径）。上游输入：已确认的 spec.md、组件根下既有 `docs/component-design.md`（或团队覆盖路径）。作者侧标准见 `hf-design`。
>
> 核心怀疑：**拿着这份设计，实现者还需要「发明」什么吗？复杂度配得上规格吗？**

## 检查项

### 契约完整（不过 = critical）

- [ ] 每个对外接口六项齐全：输入前置/输出后置/错误语义/副作用/并发时序/兼容性
- [ ] 错误模型三件事已定：错误分类与策略、传播策略（哪层翻译哪层处理）、失败路径状态保证
- [ ] 没有「错误处理：返回错误码」式的一笔带过；没有「出错后状态未定义」
- [ ] 跨边界数据的分配/释放/存活期已明确

### 结构与复杂度

- [ ] 每个模块职责一句话说得清；按变化理由组织，无明显的霰弹式修改预埋
- [ ] 依赖方向单向；无新增循环依赖；公共头文件无实现泄漏
- [ ] 每个抽象/间接层/配置项指得出真实用例或真实变化轴；无单实现接口、无「以后可能需要」
- [ ] SOLID 自检落到具体设计结论：SRP/OCP/DIP 问题有变化理由与依赖证据；LSP/ISP 问题有契约或调用方证据
- [ ] 无调用方知识泄漏：调用方不依赖无关接口、内部字段、私有宏或隐式调用顺序
- [ ] 反向检验：有没有更简单的结构也能满足规格？有 → critical finding

### 与上游一致

- [ ] 设计覆盖 spec 全部范围；没有悄悄做了 spec 之外的事
- [ ] 设计级别判定正确：触及对外接口/依赖/状态机时，组件设计草稿已先行修订并经模块架构师确认；工作项设计只引用组件基线，未重新定义组件级架构
- [ ] `modify`/`remove` 需求的兼容策略在设计中落地
- [ ] `traceability.md` 的组件设计章节/设计章节/测试设计用例列已填且与文档一致（断链 = critical）
- [ ] spec Acceptance 与 design Case ID 双向可追溯：每条 Acceptance 至少映射到一个 Case ID；每个 Case ID 都能回指 spec 需求/Acceptance；无孤儿用例或漏测验收

### 方案取舍

- [ ] 有真实可选方案时有对比（改动范围/复杂度/兼容/回滚/维护）且有推荐与理由
- [ ] 单方案写了「为什么不是更简单的 X」
- [ ] 无伪选项（同一方案三种措辞、陪跑方案）

### 测试设计（不过 = critical）

- [ ] spec 每条验收标准映射到用例；FR 有正向+异常；NFR 的 Response Measure 有可量化用例
- [ ] Case ID 汇总表是 canonical 测试设计入口；子表只展开既有 Case ID，不产生无法追溯的新用例
- [ ] modify 有回归用例；remove 有删除后语义用例
- [ ] 每个用例有层级与 mock 边界；mock 只在真实边界（硬件/外部组件/时钟），不 mock 内部纯逻辑
- [ ] 没有写不出用例的需求残留（有 → 规格不可测试，打回 `hf-specify`）

## Verdict 指引

- 契约/测试设计 critical → `需修改`；结构方向性错误（边界划错、复杂度失控）→ `重新设计`
- 发现规格本身的漏洞 → finding 标注上游问题，verdict `重新设计` 并指向 `hf-specify`
