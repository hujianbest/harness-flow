# 规格 (Specify)

目标:产出一份让后续阶段**不需要猜**的需求规格——做什么、为什么做、做到什么程度算完成。规格只回答 WHAT,不回答 HOW;出现类名、接口、表结构等实现词汇即是越界。

## 流程

### 1. 读上下文

读用户请求、相关既有代码/文档、`features/` 下是否已有相关特性目录。新特性则创建 `features/<NNN>-<slug>/` 与 `progress.md`(格式见 `hf-workflow`)。

### 2. 澄清

对照以下覆盖面,找出**会改变范围、验收标准或关键边界**的未知项:

1. 问题与动机:为谁解决什么问题,不做会怎样
2. 目标与成功标准:完成后如何判断"做对了"
3. 范围:这一轮做什么、明确不做什么
4. 核心行为:关键流程、输入输出
5. 边界与失败路径:非法输入、异常、边界值时应发生什么
6. 约束:技术栈、兼容性、性能、安全、外部依赖

**把阻塞性问题合并成尽量少的轮次一次问完**,不要逐条追问。可以合理假设的次要项,采用"假设 X,如不对请指出"的方式写入规格待用户确认,不单独提问。

### 3. 写规格

按 `references/spec-template.md` 写 `features/<NNN>-<slug>/spec.md`。核心要求:

- **功能需求逐条编号** `FR-1`、`FR-2`……每条描述一个**可观察的行为**,附至少一条 Given/When/Then 格式的验收标准。
- **非功能需求编号** `NFR-1`……必须有可判定的阈值("P95 < 200ms"),禁止"足够快""体验好"。
- **范围外显式列出**,防止实现阶段范围蔓延。
- **开放问题分级**:阻塞性问题必须在送评审前解决;非阻塞的标注后保留。
- 模糊量词(大量、快速、稳定)全部量化或删除。

### 4. 自检后送评审

对照 `skills/hf-review/references/spec-checklist.md` 自检一遍,然后按 `hf-review` 派发规格评审。评审返回"需修改"时,只修 findings 指出的问题,再送评审。

## 红线

- 从用户一句话直接跳到设计或代码
- 需求里混入实现方案(框架选型、接口签名、存储结构)
- 验收标准写成"功能正常工作"这类不可判定的句子
- 阻塞性开放问题未解决就送评审
- 把用户没确认的假设当成已确认事实写入正文
