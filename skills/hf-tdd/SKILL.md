# 测试驱动实现 (TDD)

目标:让每一行实现代码都由一个**先失败后通过**的测试拉动,交付时不留"看起来写完了但没验证"的功能。

前提:`design.md` 已评审通过且用户已确认。从其任务清单锁定**首个未完成任务**;同一时间只做一个任务。

## 铁律

```
没有失败的测试,就不写实现代码
```

先写了实现?删掉,从测试重新开始。不保留"作参考"、不"边写测试边改它"。例外仅限纯配置、纯文档类改动,且要在 `progress.md` 说明。

## 单任务循环

对当前任务依次执行:

### 1. 测试设计

动手前列出该任务要验证的行为清单:正向路径、边界值、失败路径,以及预期输入输出。对照设计的测试策略确认分层。mock 只允许出现在真正的外部边界(网络、时钟、文件系统),不允许 mock 被测对象。

### 2. RED — 写失败的测试

- 一次写一个最小测试,测试名描述行为(如 `rejects_empty_input`)
- **运行它,确认失败**,且失败原因是"行为缺失",不是编译错误或拼写错误
- 说不清"为什么预期它失败"就说明还没想清楚要实现什么,回到测试设计

### 3. GREEN — 最小实现

- 写让测试通过的最少代码,不顺手加功能、不顺手重构
- 运行测试确认转绿;**证据必须来自当前会话的真实运行**,不引用旧结果

### 4. REFACTOR — 保持绿的前提下清理

- 消除刚写代码引入的重复、坏命名、临时结构;每次小步清理后重跑测试保持全绿
- 只清理本任务触碰的范围;需要跨模块的结构性调整时,停下来告知用户,不擅自扩大改动面
- 不引入设计未声明的新抽象层

### 5. 收尾本任务

- 跑一遍全量相关测试套件,确认没有破坏既有行为
- 在 `design.md` 任务清单勾选该任务,在 `progress.md` 记录:任务 ID、测试命令、结果摘要
- 取下一个未完成任务,重复循环

## 全部任务完成后

跑全量测试套件确认全绿,然后按 `hf-review` 派发代码评审(checklist: `code-checklist.md`)。评审返回"需修改"时:只修 findings;若修复改变了行为预期,先补/改测试再改实现,仍走 RED→GREEN。

## 红线

- 先写实现再补测试,或测试一次都没运行就写实现
- 测试一写就绿(说明它没有验证新行为,重写)
- 为了让测试通过而弱化断言、跳过用例、在测试里复刻实现逻辑
- 并行推进多个任务,或跳过任务清单顺序而不说明理由
- 用上次会话/CI 的旧结果充当本次证据
- 任务做完不更新任务清单与 `progress.md`
- 实现中发现设计有错 → 停下回 `hf-design` 修订并重新评审,不"先按自己的想法改了再说"
