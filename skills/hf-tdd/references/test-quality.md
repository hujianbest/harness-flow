# 测试质量判据

> 配套 `hf-tdd`（编写时内建质量）与 `hf-review` 的 R3 测试评审（独立检查）。原则与语言/框架无关；按工作项语言套用对应 `*-coding-standards`。

## 断言强度

测试的价值 = 它能抓住的错误实现的数量。按下表升级断言：

| 弱断言 | 问题 | 强断言 |
|---|---|---|
| `assert result is not None` | 任何非空垃圾都能过 | 断言 `result` 具体字段值 |
| `assert f(x) == OK` | 只验证返回码，副作用全部漏掉 | 返回码 + 状态变化 + 事件/输出各一条断言 |
| `assert not raises(f())` | 「没崩」不等于「做对了」 | 断言 f 的具体结果 |
| `assert count > 0` | 1 和 1000 都能过 | `assert count == 3`——期望值是确定的就写确定值 |
| 末尾 `assert ok` | ok 的来源经过多步，失败时无信息 | 在每个判定点直接断言具体值 |

**Mutation 自检**：写完测试问自己——把实现里的 `<` 改成 `<=`、把错误返回删掉、把事件投递注释掉，这个测试会失败吗？任何一个「不会」都意味着断言有洞。

**副作用必须显式断言**。一个操作的可观察结果通常有三类：返回值、状态变化、对外输出（事件/日志/写硬件）。spec 的 Then 里提到的每一项都要有对应断言，包括**负向断言**（「不发出事件」要断言队列计数为 0）。

## 测试命名

格式：`<被测对象>_<场景>_<预期结果>` 或等价的可读句子。读测试名就能知道坏的是什么行为。

```text
✅ test_set_mode_rejects_invalid_mode_without_state_change
✅ test_config_loader_falls_back_to_defaults_when_crc_fails
❌ test_1 / test_set_mode / test_error          （失败时毫无信息）
❌ test_set_mode_works_correctly                （"correctly"是什么？）
```

测试名含「And」连接两个独立行为 → 拆成两个测试。

## 一个测试一个行为

- 一个用例只有一组 Given/When/Then；多个 When 串联 = 多个测试
- 测试体内出现 if/switch 给不同分支不同断言 → 拆成参数化测试或独立用例
- Arrange-Act-Assert 三段清晰可辨；Act 通常只有一行

## Fixture 与 setup

- fixture 只放**所有**用例共享的最小公共状态；个别用例需要的前置放该用例里
- setup 超过 ~15 行或含分支 → 提取命名清晰的辅助函数（`given_service_in_safe_mode()`）
- 禁止用例间共享可变状态：每个用例从干净状态开始，顺序无关、可单独运行
- 不依赖真实时间（实时性 NFR 测量除外）：注入受控时钟；测试里出现 `sleep` 是 flaky 信号

## Mock / Stub / Fake 边界

| 依赖类型 | 处理 |
|---|---|
| 硬件、寄存器、外设 | fake/仿真层（HarnessFlow 推荐 fake 优于 mock：fake 有真实行为，测试断言结果而非调用） |
| 外部组件 / 跨进程服务 | mock 或 fake，按接口契约打桩 |
| 时钟、随机数 | 注入受控实现 |
| 文件/网络等慢速 IO | fake 或内存实现 |
| **本模块内部函数 / 纯逻辑** | **不 mock**，直接走真实代码 |

红线：

- 不为可测性给生产类加 public test-only 方法；需要观察内部状态 → 通过公共行为观察，或说明设计问题回 `hf-design`
- 不以 `mock.assert_called_once()` 替代行为断言——验证「调用过」只锁死实现方式，验证「结果对」才锁住行为。交互本身就是契约的场景（如「失败时必须上报诊断事件」）除外
- mock 行为与真实依赖的契约不符（如真实函数会返回 ERR_BUSY 但 mock 永远 OK）= 测试在验证一个不存在的世界

## 测试代码也是代码

测试代码同样适用 `hf-clean-code`：命名表意、无复制粘贴的大段重复（提取辅助函数）、无注释掉的死测试。但有一条差异：**测试宁可略有重复也要可读**——读一个测试不应需要跳转三层辅助函数才能知道它在验证什么。

## 评审视角速查（供 hf-review R3 测试评审使用）

- 每条验收标准 / 测试设计用例都有对应测试；modify 有回归用例，remove 有删除语义用例
- 每个 NFR 的 Response Measure 有可量化验证（latency 直方图、size 输出、leak 报告等）
- RED 证据存在：测试曾经失败且失败原因正确
- 抽查 2-3 个关键测试做 mutation 自检
- 无永远成功的测试（无断言、断言恒真、只靠 printf 人工观察）
- 无顺序依赖、无未受控的时间/随机/全局状态
