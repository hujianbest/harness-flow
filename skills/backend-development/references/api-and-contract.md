# API 与契约详表

> `backend-development` 的 **API 契约** 维度深表。SKILL.md 承载红线，本文承载可直接对照的约定与清单。接口是组件间契约：可观察的接口变更走 `hf-specify` 的接口变更纪律 + 基线，并列已知消费者与兼容策略。

## 资源建模

- 用名词复数表达资源集合，层级表达从属关系：`/orders`、`/orders/{id}`、`/orders/{id}/items`。
- 动词不进 URL；用 HTTP 方法表达动作。确需"动作"语义（如 `cancel`）时收敛为子资源或显式命令端点并在契约中说明。
- 集合返回包裹在稳定信封里（`{ "data": [...], "page": {...} }`），便于无破坏地加分页/元数据。

| 方法 | 语义 | 幂等 | 典型成功码 |
|---|---|---|---|
| GET | 读取，不产生副作用 | 是 | 200 |
| POST | 创建 / 触发非幂等动作 | 否 | 201（带 `Location`）/ 202 |
| PUT | 整体替换 | 是 | 200 / 204 |
| PATCH | 局部更新 | 视实现 | 200 / 204 |
| DELETE | 删除 | 是 | 204 |

## 状态码语义

| 码 | 用于 | 不要用于 |
|---|---|---|
| 200 | 成功且有响应体 | 失败结果（不要"200 包错误"） |
| 201 | 创建成功，返回新资源/`Location` | 异步受理（用 202） |
| 202 | 已受理，处理异步进行 | 同步完成 |
| 204 | 成功且无响应体（删除/更新） | 需要返回体时 |
| 400 | 语法/请求格式错误 | 业务校验失败（用 422） |
| 401 | 未认证（缺/失效凭证） | 已认证但无权（用 403） |
| 403 | 已认证但无权限/越权 | 资源不存在（用 404 或为防枚举用 404 替代） |
| 404 | 资源不存在 | 校验失败 |
| 409 | 与现有状态冲突（重复、版本冲突） | 普通校验 |
| 422 | 语义校验失败（字段级） | 认证/授权问题 |
| 429 | 超出限流，配 `Retry-After` | 服务端自身错误 |
| 5xx | 服务端错误，**不泄漏内部细节** | 客户端错误（4xx） |

## 错误信封

统一的机器可读信封，字段稳定、可演进（参考 RFC 7807 思路）：

```jsonc
// 字段级校验失败
HTTP 422
{ "error": {
    "code": "validation_failed",
    "message": "Validation failed",
    "fields": { "email": "required", "age": "must be >= 0" },
    "request_id": "req_01H..."
} }

// 资源不存在
HTTP 404
{ "error": { "code": "order_not_found", "message": "Order not found", "request_id": "req_01H..." } }
```

- `code` 是稳定的机器标识（消费者据此分支），`message` 面向人可变。
- 5xx 只回笼统信息 + `request_id` 供排障，**绝不**回堆栈/SQL/内部路径。
- `request_id` 贯穿日志与响应，把客户端报错与服务端日志关联起来。

## 分页 / 过滤 / 排序

- **分页**：小数据集可用 `?page=&page_size=`；大数据集/高写入用游标 `?cursor=&limit=`（避免 offset 漂移与深翻页慢）。响应回 `next_cursor`/总数（总数昂贵时可省）。
- **过滤**：白名单可过滤字段，未知字段拒绝（400）而非静默忽略。
- **排序**：`?sort=-created_at,name`（`-` 表降序）；只允许白名单字段，且这些字段要有索引（见数据访问维度）。
- 默认值与上限：`limit` 有服务端上限（如 ≤100），防止"一把拉全表"。

## 版本与弃用

- 破坏性变更（删字段、改类型/语义、收紧校验、改默认值）走新版本（URL `/v2` 或媒体类型），不在原版本上原地破坏。
- 弃用流程：标注弃用 → 文档/响应头告知（`Deprecation`/`Sunset`）→ 通知已知消费者 → 过渡期后下线。
- 加字段、加可选参数、放宽校验通常是兼容变更，但仍要确认无消费者依赖"字段不存在"。

## 契约校验证据

- OpenAPI/JSON Schema 作为单一契约来源；请求/响应据此校验。
- 契约/快照测试覆盖：每个状态码分支、错误信封结构、分页/排序边界。
- 多消费者时优先消费者驱动契约测试（Pact 等），防止服务端无声破坏下游。
