# 弹性、后台任务与生产就绪详表

> `backend-development` 的 **弹性与依赖容错 / 后台任务与异步 / 生产就绪** 维度深表。SKILL.md 承载红线，本文承载取值参考与可直接套用的模式。

## 超时

- 每个跨进程调用都要有超时，不依赖框架/客户端的"默认无限等"——无超时是连接池耗尽与雪崩的头号原因。
- 分层设置：单次调用超时 < 上游请求总预算；总预算要为重试留余量（`总预算 ≳ 单次超时 × 重试次数`，否则重试还没做完上游就已超时）。
- 区分连接超时与读取超时；流式/长轮询用独立的更长超时，不要套用普通接口的值。

| 依赖 | 起始参考值 | 说明 |
|---|---|---|
| 进程内缓存 | 亚毫秒级 | 不需重试 |
| Redis/本地网络缓存 | 50–200ms | 失败可降级到源 |
| 数据库查询 | 按查询类型分级；OLTP 常见 1–5s | 配合慢查询日志与索引 |
| 同区下游服务 | 0.5–3s | 留足重试余量 |
| 第三方外网 API | 2–10s | 抖动大，优先熔断 + 降级 |

> 值随业务 SLA、网络拓扑、p99 实测而定；以上仅为讨论起点，最终以 `hf-specify` 产出的 QAS 阈值与压测为准。

## 重试

- **只重试幂等且瞬时的失败**：网络抖动、超时、5xx、限流（429，按 `Retry-After`）。
- **不重试**：4xx（客户端错误，重试也错）、非幂等且无幂等键的写操作（重试 = 重复副作用）。
- 退避 = 指数 + 抖动（`base * 2^n + random`），避免同时重试形成"重试风暴"。设次数上限与总时间上限。
- 重试要与幂等键、超时预算协同：重试前确认整体没超总预算。

```text
attempt = 0
loop:
  try call()                      # 单次有超时
  on success: return
  on error e:
    if not isTransient(e): raise  # 4xx / 永久错误直接抛
    attempt += 1
    if attempt >= MAX: raise
    sleep(min(cap, base * 2^attempt) + jitter())
```

## 熔断与隔离

- **熔断**：对某依赖连续失败超阈值时打开熔断，一段时间内快速失败（不再发起调用），半开探测恢复后再闭合。避免把请求线程堆在已挂的依赖上。
- **隔离（bulkhead）**：为不同依赖分配独立连接池/并发额度，单个依赖被打满不拖垮全局。
- **降级**：熔断打开/依赖不可用时走降级路径——返回缓存/默认值/部分结果，或明确告知功能暂不可用，而不是整链路 5xx 雪崩。
- 证据：故障注入（杀依赖、注入延迟）下系统进入降级而非雪崩；熔断打开/半开/闭合状态可观测。

## 后台任务与异步

- **移出请求路径**：发邮件/短信、生成报表、转码、调慢/不稳定的第三方——入队，请求侧回 202/立即返回。
- **投递语义**：多数队列是"至少一次"，因此任务必须幂等（用业务幂等键或处理前查"是否已处理"）。不要假设"恰好一次"。
- **可见性超时 / 租约**：任务领取后在处理窗口内对其他 worker 不可见；处理超时则重新可见被重投——再次印证幂等的必要。
- **重试与死信**：失败任务有限次重试（退避），耗尽进死信队列（DLQ）并告警，不静默丢；DLQ 要有人/流程消费。
- **自包含**：任务从负载与持久存储读取所需上下文，不依赖请求级状态（请求早已结束）。
- **worker 隔离**：worker 与 API 进程分离部署，互不抢占资源；worker 也要优雅停机（处理完在途再退出）。

## 生产就绪

### 健康检查 vs 就绪检查

| 探针 | 回答 | 失败时编排器动作 | 实现 |
|---|---|---|---|
| liveness `/health` | 进程是否活着 | 重启容器 | 极轻量，**不**探外部依赖（否则依赖抖动会引发无意义重启） |
| readiness `/ready` | 是否能接流量 | 暂时摘出负载均衡 | 探关键依赖（DB/缓存/必需下游）；任一不通即非 200 |

### 优雅停机（SIGTERM）

```text
收到 SIGTERM
  1. readiness 转为不就绪（编排器停止转入新流量）
  2. 停止 accept 新连接 / 停止从队列领新任务
  3. 排空在途请求与在途任务（带上限超时）
  4. 关闭 DB/缓存/MQ 连接，flush 日志/指标
  5. 进程退出 0
```

不硬杀在途请求/任务；为排空设最大等待时间，超时再强制退出。

### CORS 与安全头

- CORS：允许来源用**显式白名单**，不用 `*`——尤其 `Access-Control-Allow-Credentials: true` 时 `*` 既不安全也不被浏览器接受。按需限定方法与头。
- 安全响应头（按平台选用）：`Strict-Transport-Security`、`X-Content-Type-Options: nosniff`、`X-Frame-Options`/`Content-Security-Policy: frame-ancestors`、合理的 `Referrer-Policy`。
- 强制 HTTPS；不在错误响应/响应头里暴露框架版本等内部信息。
