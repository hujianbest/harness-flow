# 设计 (Design)

目标:把已批准的规格转化为**可执行的实现方案**,让 TDD 阶段照着做而不需要临场做架构决策。设计的最后一节是任务清单——HarnessFlow 不设独立的任务阶段,拆分是设计的收尾。

前提:`spec.md` 已评审通过且用户已确认。否则先回 `hf-specify`。

## 流程

### 1. 读规格与代码现状

通读已批准的 `spec.md`,提取:功能需求、NFR 阈值、约束、范围外。再读现有代码中将被触碰的部分:目录结构、既有模式、可复用模块、测试组织方式。**设计必须顺着现有代码的约定走**,除非有明确理由偏离并写入决策记录。

### 2. 关键决策:至少两个方案

对每个影响后续实现的决策(架构形态、模块划分、技术选型、数据存储方式……):

- 列 ≥2 个可行方案,写清各自的取舍(复杂度、性能、维护成本、与现有代码的契合度)
- 给出选择与理由,理由要能被评审者冷读检验
- 遵守 YAGNI:只为规格中已确认的需求设计,不为想象中的未来需求引入抽象层

### 3. 写设计文档

按 `references/design-template.md` 写 `features/<NNN>-<slug>/design.md`,覆盖:

- **架构总览**:模块划分与依赖方向(复杂时配 Mermaid 图)
- **接口与数据契约**:模块间接口签名、数据结构、错误模型——写到实现时不需要再发明的程度
- **NFR 落点**:规格里每条 NFR 由哪个机制满足、如何验证
- **错误处理**:关键路径的失败模式与对策
- **测试策略**:测试分层(单元/集成/端到端)、测试数据、如何跑

### 4. 任务拆分

把设计拆成有序任务清单,写入 `design.md` 最后一节。每个任务必须:

- **小**:一次 TDD 循环内可完成(一个行为单元,通常 ≤ 半天工作量)
- **可独立验证**:有明确的"完成即测试通过"判据
- **可追溯**:标注覆盖哪些 FR/NFR;所有 FR/NFR 必须被至少一个任务覆盖
- **有序**:按依赖排列,优先打通最薄的端到端路径(walking skeleton),再填充

### 5. 自检后送评审

对照 `skills/hf-review/references/design-checklist.md` 自检,按 `hf-review` 派发设计评审。返回"需修改"时只修 findings,再送评审。

## 红线

- 规格未批准就开始设计
- 只写一个方案,没有对比就"选定"了技术
- 设计里出现规格没有的新需求(发现规格缺漏 → 停下,回 `hf-specify` 修订规格)
- 任务粒度大到无法一次 TDD 循环完成,或没有映射到需求编号
- 为"未来可能用到"引入接口、抽象层、配置项
- 忽略现有代码的既有模式,平行发明一套新结构
