# HarnessFlow 收尾（Ship）

## 总览

收尾只回答两个问题：**这个工作项真的可以关闭吗？关闭之后给仓库留下什么？**

对应两个动作，缺一不可：

1. **完成核验**：对照 Definition of Done 逐项检查证据。"评审通过了"不等于"可以关闭"——DoD 是人在最后把关时的固定核验清单，防止靠印象关单。为什么把核验钉死在最后一关：前面所有阶段的"通过"都只证明某一层达标（spec、design、test/code），没有任何一个节点核对"全部条件是否同时满足、链路是否闭合"。这最后一道人工核验是对整个工作项的收口。
2. **资产沉淀（promotion）**：把组件根下 `features/<id>-<slug>/`（或团队 `AGENTS.md` 覆盖路径）里**已确认**的规格与设计，按原模板结构沉淀为同一组件根下 `docs/` 的长期文档。promotion 不是重新创作交付件；原 spec / design / component-design 模板中的正文、章节、表格和锚点默认保留，只做最小清理。过程工件是给本次开发用的；长期资产是给下一个工作项、下一个人、下一轮 AI 用的。不做 promotion，工件就停在 `features/` 里腐烂，组件设计基线逐渐失真——下一个工作项会基于过时的长期文档做设计，漂移就此开始。

收尾**不修代码、不补测试、不写实现**：核验发现缺口 → 回对应阶段补，补完再回来。收尾是收口，不是兜底。把"顺手把缺口补了"当成高效，会绕过该阶段的独立评审与证据记录，破坏角色分离。

> 本节自检：[ ] 明确这两个问题不可合并（不能跳过核验直接写 closeout）；[ ] 识别出任何"我来顺手补一下缺口"的冲动都属于破坏信任而非效率。

## 工作流

### 1. DoD 核验

对照 `references/definition-of-done.md` **逐项**检查，每项写一个结论（满足 / 缺口 + 去向）。逐项的硬要求：DoD 表里的每一行在 closeout 里都必须有一个对应的核验结论，"整体看起来都满足"不算逐项。

任一项不满足，按下表分流回去，不在收尾阶段就地修：

| 缺口 | 回到 | 在 closeout 里怎么写 |
|---|---|---|
| 需求条目无对应通过测试 / 验收标准未覆盖 | `hf-tdd` | `缺口 → hf-tdd`，不写"已确认" |
| `reviews/` 缺某个门禁（R1/R2/R3）的评审记录，或 critical/important findings 的 Resolution 列有空缺 | `hf-review`（补评审）或对应阶段返工并回填 Resolution | `缺口 → hf-review / 返工` |
| `plan.md` 门禁状态表与 `reviews/` 实际记录不一致 | 修正门禁表；状态造假按 critical 处理，停止收尾 | 不写结论，先修状态 |
| 实现与 `design.md` / 组件设计漂移 | `hf-design`（改工件）或 `hf-tdd`（改代码），漂移修复后受影响 R 门禁重做 | `缺口 → hf-design / hf-tdd` |
| 证据缺失或过期（RED/GREEN 证据行、静态分析输出不是本轮真实产生的） | `hf-tdd` 补真实证据，不补叙述 | `缺口 → hf-tdd（补证据）` |
| 适用约束审计只写 `clean` 没有证据引用 | `hf-tdd`（REFACTOR 证据）/ `hf-review`（finding/resolution）/ 静态分析 | `缺口 → 补证据引用` |

**缺口不闭环就不写 closeout**：把缺口"解释掉"是最常见的关单失败。证据必须是真实命令输出、commit 锚点、评审记录路径、债务登记，不是叙述。

> DoD 核验自检：[ ] DoD 每一行都有显式结论（不是整体打勾）；[ ] 每个缺口都映射到了具体阶段而非被解释；[ ] 任一 `critical-open` 约束或门禁造假都已停止收尾。

### 2. 追溯终验

通读同一组件根/工件根下 `features/<id>-<slug>/traceability.md`（或团队覆盖路径）：每条需求 → 设计章节 → 测试用例 → 代码/测试文件 → 证据的链路必须闭合；每个 `N/A` 必须写出理由。

**断链 = spec-design-code 不一致的直接信号**，按 critical 处理：不要把"没填"标记成 `N/A` 仓促过关。链路中间任意一格空缺（缺测试、缺证据、缺代码文件引用），都属于断链，回对应阶段补真实内容。

判定规则：

| traceability 行的实际情况 | 收尾处理 |
|---|---|
| 链路每一格都填了且指向真实工件 | 通过，写结论 |
| 某一格空缺但写了 `N/A` + 理由（如"纯文档需求无代码""验收标准合并测试"） | 通过，理由可冷读 |
| 某一格空缺，标记 `N/A` 但**无理由** | 断链，按 critical 处理，回阶段补 |
| 链路填了但指向过期或不存在的文件/证据 | 断链，按 critical 处理 |

> 追溯终验自检：[ ] 每行链路每一格都有内容或带理由的 `N/A`；[ ] 没有"无理由 N/A"；[ ] 引用的文件/证据路径存在且属本轮。

### 3. Promotion

按 `references/promotion-checklist.md` 同步长期资产。promotion 的核心纪律是**保留原模板、做最小清理**——已确认的规格、设计和组件设计正文是交付件，过程内容留在 `features/`。

| 过程工件 | 长期资产 | 条件 |
|---|---|---|
| `spec.md` | 组件根下 `docs/ar-specs/<id>-<slug>.md`（或团队覆盖路径） | AR / CHANGE 工作项**必做**；纯缺陷（无规格变更）`N/A` |
| `design.md` | 组件根下 `docs/ar-designs/<id>-<slug>.md`（或团队覆盖路径） | 有正式工作项设计时**必做** |
| `component-design-draft.md` | 组件根下 `docs/component-design.md`（或团队覆盖路径） | 本工作项修订了组件设计时**必做**（模块架构师确认后） |

**只 promote 已通过评审（verdict 通过、findings Resolution 闭环、attended 模式人工确认）的工件**。promotion 不是引入未批准变更的后门：长期文档里的对外接口、错误码集、状态机变化，必须与 spec 中批准的 `modify`/`remove` 基线一一对应。

必须做的清理（去掉草稿专属内容，保留交付件主体）：

- **去掉**：Open Questions 节（保留已闭合结果，不留待决项）、评审 findings 应答、过程笔记、会议纪要片段、`TODO` / `待澄清` / 模板提示残留。
- **保留**：原模板内容——需求 / 设计正文、接口契约表、错误模型、测试设计表、组件设计章节、编号与表格结构、追溯锚点（工作项 ID、上游来源、测试设计用例 ID、评审记录路径）。
- **补全**：长期文档头部记 Promoted From（指向过程工件的 commit 锚点）；长期文档已有「变更记录」表时追加本次修订 `(日期, 触发工作项, 摘要)`，没有该表时不为补表而重排全文。

最小清理的正反例（promotion 是清理，不是重写）：

```text
✅ 去掉 spec.md 的 Open Questions 与评审应答，保留需求条目、验收标准、
   接口候选契约表、追溯锚点 → docs/ar-specs/<id>-<slug>.md
✅ 组件设计只更新本次受影响章节（状态机一节），
   其他章节原样保留 → docs/component-design.md
❌ 把 design.md 改写成一段摘要，丢掉接口契约表、错误模型、测试设计表，
   理由是"长期文档不该这么细"
❌ promotion 时顺手重排组件设计全文、统一术语、改写未受本工作项影响的章节
❌ promotion 时"顺手"改了一个错误码名字（未经 spec modify 流程的变更走私）
❌ 修订了组件状态机，closeout 写"已修订"但 docs/component-design.md 没动
❌ 只 promote design 不 promote spec，长期库里设计和规格断链
```

组件设计修订的额外纪律：只更新本次受影响的章节，不顺手重排或"统一"其他章节。

> Promotion 自检：[ ] 只 promote 已评审通过的工件；[ ] 去掉了 Open Questions / 过程笔记 / 评审应答，保留原模板章节/表格/锚点；[ ] 变更记录按规则追加（不强行新建）；[ ] 组件设计只动受影响章节。

### 4. Closeout 记录

写同一组件根/工件根下 `features/<id>-<slug>/closeout.md`（或团队覆盖路径，一页内）。closeout 是结构化收尾记录，不是模糊总结。必须包含：

- **DoD 核验结果摘要**：每项的满足/缺口结论（不是"整体通过"）。
- **promotion 路径表**：同步了什么工件到什么长期资产路径；`N/A` 项写理由。
- **遗留债务清单**：每条带去向（新工作项编号 / 已登记的 issue 链接）；"后续优化"没有去向 = 未登记债务。
- **复盘一句话**：本次流程哪里最磨损，反哺哪个 skill 或模板。

closeout 模板见 `references/closeout-template.md`。

> Closeout 自检：[ ] DoD 摘要是逐项结论而非整体；[ ] promotion 路径表完整（含 N/A 理由）；[ ] 每条债务有去向；[ ] 复盘一句话已写。

### 5. 人确认关闭

把 closeout 呈给人。人确认后工作项关闭；组件根下 `features/<id>-<slug>/`（或团队覆盖路径）**原地保留**（不移动、不删除），保证追溯链接长期有效。

`unattended` 模式下：DoD 核验、promotion、closeout 照做不省，critical 阻塞照样阻塞；只是不在此处停顿等人确认，关闭后把 closeout 与全部 `reviews/` 呈给人做事后审计。unattended 移除的是人工停顿，不是 DoD 或 promotion。

> 关闭自检：[ ] attended 模式呈人确认后才关闭；[ ] `features/` 目录原地保留。

## 风险信号

- 跳过 DoD 直接写 closeout（"评审都过了肯定没问题"）
- promotion 过度改写原模板正文、重排章节或压缩表格，导致长期交付件丢失规格 / 设计模板内容
- promotion 把过程内容原样带入长期文档（Open Questions、过程笔记、评审应答）
- 只 promote 设计不 promote 规格，或修订了组件边界却没同步 `docs/component-design.md`
- promote 了未评审通过的工件，或长期文档里的接口/错误码与 spec 批准基线不一致
- 遗留债务写"后续优化"却没有去向（没有新工作项、没有登记）
- 把"补证据"做成"补叙述"——证据必须是真实命令输出与 commit/记录锚点
- `unattended` 模式下省略 DoD 或 promotion（连续执行不等于放松质量）

## 合理化反驳

| 话术 | 现实 |
|---|---|
| 「评审都过了，DoD 走个形式就行」 | DoD 是对整个工作项的收口核验，前面各阶段只证明单层达标，没有任何节点核对"全部条件同时满足"。跳过它 = 靠印象关单 |
| 「这个缺口我顺手补一下更快，不用回阶段」 | 收尾是收口不是兜底。就地补会绕过该阶段的独立评审与证据记录，破坏角色分离。缺口回对应阶段 |
| 「长期文档改写得更简洁更好读」 | promotion 是保留模板的最小清理，不是重写。已确认的接口契约表/测试设计表是交付件，改成摘要就丢了规格 |
| 「只 promote design 就行，spec 大家都记得」 | 长期库里设计和规格断链，下一个工作项会基于过时文档设计。AR/CHANGE 工作项 spec 与 design 都要 promote |
| 「债务先记一句'后续优化'，到时候再说」 | 没有去向的债务 = 未登记债务。"到时候"永远不会到。必须落到新工作项或登记的 issue |
| 「这条 traceability 行没填测试，标 N/A 收尾吧」 | 无理由的 N/A 是断链。断链是 spec-design-code 漂移信号，按 critical 处理回阶段补 |
| 「unattended 模式，promotion 省了吧」 | unattended 只移除人工停顿，不移除任何质量动作。DoD、promotion、closeout 一项不少 |
| 「证据我口头确认过，不用再贴命令输出了」 | "我已经验证过了"不构成证据。RED/GREEN 证据行、commit 锚点、评审记录路径才构成 |

## 微小修改与缺陷工作项的 DoD 裁剪

裁剪的是文档量，永远不是质量门槛。裁剪规则（来自 `using-hf` §6，DoD 细则见 `references/definition-of-done.md` 的裁剪说明）：

| 工作项类型 | 可裁剪 | 不可裁剪 |
|---|---|---|
| 微小修改（几行、无接口变化、风险低） | 第 1-3 项（spec/design 评审）替换为「`plan.md` 中的验收标准已全部有通过测试」；第 8 项设计一致性随之一并简化 | 第 4-7 项（证据行、测试通过、完整套件、R 门禁记录）、第 10 项（静态分析）、第 11 项（traceability）**不降低** |
| 缺陷工作项（DTS，来自 `hf-fix`） | 第 1-3 项替换为「`fix.md` 的复现/根因/修复边界完整，复现测试先失败后通过」 | 其余各项**不降低** |
| 任何情况 | —— | 证据要求（4-7、10）与一致性要求（8、11）不可裁剪。紧急不等于绕过 |

> 裁剪自检：[ ] 裁剪只动了文档量；[ ] 证据与一致性要求未降；[ ] 微小/缺陷工作项用了对应替换项而非整体跳过。

## 验证清单

收尾完成前逐项确认：

- [ ] DoD **逐项**有结论；缺口已返工闭环而不是被解释掉
- [ ] traceability 链路闭合；每个 `N/A` 有理由；无"无理由 N/A"
- [ ] 应 promote 的资产已保留原模板主体并完成最小清理；只 promote 了已评审通过的工件；变更记录已追加
- [ ] closeout.md 已落盘：逐项核验摘要、promotion 路径表、债务去向、复盘一句话
- [ ] 微小/缺陷工作项按裁剪规则处理，质量门槛未降
- [ ] 人已确认关闭（attended 模式）；`features/` 目录原地保留；`unattended` 模式下 DoD/promotion/closeout 未省
