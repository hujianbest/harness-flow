# HarnessFlow 缺陷修复

## 总览

修缺陷最大的失败模式不是修不好，而是**修了症状**：没复现就开改、把"不再报错"当"修好了"、顺手把周围看不顺眼的代码一起动了。失败一旦发生，根因从未被真正理解，缺陷会以变种形式反复回来。纪律一句话：

> **先复现，再归因，后修复；没有失败的测试就没有修复。**

为什么这个顺序不可妥协：跳过复现就改，等于猜——改完"不报错"可能只是把触发路径挪走了、或是碰巧没踩到，根本原因还在。跳过归因直接改现象，症状被压住但根因仍在，下一个变种会回来。没有先失败的测试就声称修好了，等于"我手动验了一下"——不可复现、不可回归、下次改动不会自动重跑。先复现→归因→修复的顺序，把"修好了"从主观感觉变成可执行、可回归的事实。

产出按问题严重度伸缩：简单缺陷把三段记录（复现 / 根因 / 修复边界）写进组件根下 `features/DTS<id>-<slug>/fix.md`（或团队 `AGENTS.md` 覆盖路径）即可；根因涉及设计缺陷或行为变更时，回到完整的 spec/design 流程。`fix.md` 模板见 `references/fix-record-template.md`。

缺陷不走完整主流程，而是从修复点接回正轨（来自 `using-hf` §5 旁路）：

```text
hf-fix → hf-tdd → R3(hf-review) → hf-ship
```

`hf-fix` 产出 `fix.md`（复现/根因/修复边界）后，**修复实现回到 `hf-tdd`**：先写一个复现缺陷的失败测试（RED），再最小修复让它转绿。修复后的测试与代码同样经 R3 评审，收尾同样经 `hf-ship` 做 DoD 核验。hotfix 可以压缩文档，但不能跳过任何质量门槛（见末节）。

> 本节自检：[ ] 把三段顺序当成不可妥协的纪律；[ ] 识别"先改一行试试看"为破坏信任而非高效；[ ] 修复实现走 TDD、收尾走 R3+ship，不是修完就结束。

## 工作流

### 1. 复现

用最小步骤稳定复现，在 `fix.md` 记录：触发输入、环境（版本/分支、平台/板卡、配置）、预期 vs 实际、证据（日志 / core / trace 路径）、复现稳定性。

**复现不了就不要修。** 这是铁律，不是建议。无法复现的修复都是盲修补丁——你不知道改的是不是真因，也不知道改完是否真的不再发生。

| 复现情况 | 处理 |
|---|---|
| stable（稳定复现） | 继续 step 2 |
| flaky（偶发） | 先提高复现率：加压、注入延迟、缩小区间、提高采样；记录已尝试的手段与频率 |
| 不可复现 | **不修**。记录已排除的假设与证据，交回报告人补信息（更精确的触发条件、环境、日志），不上"盲修补丁" |

```text
✅ flaky 问题：注入延迟/加压把复现率从 1% 提到 30%，记录手段，继续定位
❌ flaky 问题：加个重试让它"不再出现"，不解释机理，标记为"已修复"
❌ 不可复现：猜一个最可能的点打补丁，赌它生效
```

> 复现自检：[ ] 有最小复现步骤与环境记录；[ ] 证据可回看（日志/core/trace 路径）；[ ] flaky 写了已尝试的复现手段与频率；[ ] 不可复现的不上盲补丁，交回报告人。

### 2. 根因分析

从现象向后追因果链，**每一步要有证据**（日志、代码路径、git 历史、二分），直到回答「**为什么会发生**」而不是「在哪发生」。连续追问"为什么"直到再往下是设计决策或外部事实为止。

区分三层并全部写进 `fix.md`：

| 层 | 回答的问题 | 反例（不合格的根因） |
|---|---|---|
| 直接原因 | 哪行代码 / 哪个配置错了 | "空指针访问"——只说了"在哪发生"，没说为什么会发生 |
| 根本原因 | 为什么会写错 / 漏掉——契约不清？边界没定义？测试缺口？ | "开发者疏忽"——不可操作，无法防止下次 |
| 波及范围 | 同样的模式还出现在哪里 | 只修了这一处，同类隐患继续潜伏 |

**被排除的假设要记录排除证据**——这是防止下一个人重走弯路的最廉价文档。如果第二个人来复查，他脑子里会冒出你已经排除的假设；写下来就帮他省掉一次重复劳动。

```text
✅ 因果链：现象(panic) → 解引用前未判空(直接原因) → 契约未声明该指针可能为空，
   调用方按"必非空"假设(根本原因) → 检索发现 3 处同样假设(波及范围)
✅ 排除假设：| 假设并发竞态 | 二分到单线程仍复现，排除 |
❌ 根因写"空指针" → 这是直接原因，不是根因；没回答"为什么会漏判空"
```

**改一行试试看 → 不行再改一行——这是猜，不是归因。** 两次"试试看"不中就必须停下来回到证据：重新读日志、做二分、加诊断输出，而不是继续盲改。

> 根因自检：[ ] 因果链每步有证据；[ ] 直接/根本/波及三层齐全；[ ] 根本原因可操作（能防止下次，不是"疏忽"）；[ ] 已排除的假设记录了排除证据。

### 3. 界定修复边界

明确写下三件事，全部进 `fix.md`：

- **最小安全修复范围**：哪些文件 / 函数会改
- **显式不修什么**：周边坏味道、同类风险点——登记去向（新工作项 / issue），不在本次顺手修
- **回退策略**：怎么回退

然后按根因性质分流：

| 根因性质 | 路径 |
|---|---|
| 实现错误（逻辑 / 边界 / 资源），契约本身正确 | 直接进入 step 4（`hf-tdd` 修复） |
| 行为 / 契约需要变更（错误码、接口语义、阈值） | 这是**变更不是修复** → 走 `hf-specify`（`modify` 条目 + 基线） |
| 设计缺陷（边界划错、并发模型错） | 回 `hf-design` 修订后再实现 |
| 波及范围里发现同类隐患 | 登记为独立工作项，不在本次顺手修 |

"显式不修"是范围纪律的核心：路过的问题登记，不顺手修。修复 diff 里混进重命名、格式化、顺手清理，会让评审者无法判读"这次修复到底改了什么"，回归出问题时也无法定位。

```text
✅ 修复边界：改 Session.validate() 一处；显式不修的 2 处同类隐患登记为 DTS<新id>
❌ 修复 diff 里混着重命名、格式化、顺手清理周边坏味道
```

> 边界自检：[ ] 最小修复范围、显式不修（带去向）、回退策略三件齐全；[ ] 行为/契约变更走了 specify 没伪装成"修复"；[ ] 波及隐患登记为独立工作项，没顺手修。

### 4. TDD 修复（回到 hf-tdd）

修复实现**回到 `hf-tdd`**，按它的 RED→GREEN→REFACTOR 纪律执行（纪律同 `hf-tdd`，这里只讲缺陷修复特有的纪律）：

- **先写复现缺陷的失败测试（RED）**：把 step 1 的复现步骤翻译成自动化用例，**确认它因为这个缺陷而失败**（失败原因是目标缺陷，不是测试本身写错或环境问题）。这个测试就是回归屏障——以后任何人改动若再次引入这个缺陷，它会立即变红。没有这个先失败的测试，就不能声称"修复"。
- 最小修复让它转绿（GREEN）；**完整测试套件**确认无回归；必要的清理留在 REFACTOR（GREEN 改行为、REFACTOR 改表达，不在一个 diff 混）。
- 根因是"测试缺口放过了它" → 补的不只是这一个用例：检查同一缺口下还有哪些边界没有测试，一并补上（登记进 `fix.md`）。

```text
✅ RED: 把复现步骤翻成 test_session_validate_null_input → FAILED: panic on null
   GREEN: 加判空 → 返回 ERR_INVALID_ARG → 该用例转绿，完整套件全绿
❌ 修复后手动验一下"不报错了"，不加任何自动化测试
❌ 加了测试但没先看它失败（测试一写就过 = 没在验证目标缺陷）
```

> TDD 修复自检：[ ] 存在先失败后通过的复现测试，失败原因是目标缺陷；[ ] 完整套件无回归；[ ] 测试缺口根因下补齐了同类边界；[ ] GREEN/REFACTOR 不混在一个 diff。

### 5. 收尾（R3 评审 + hf-ship）

`fix.md` 补全四问：

1. 根因是什么（三层）
2. 为什么现有测试没拦住（测试缺口分析）
3. 修复改了什么（diff 摘要 + commit）
4. 同类风险登记在哪（工作项 / issue 链接）

代码与测试按 `hf-review` 的 R3（test + code rubric）评审，由独立 `hf-reviewer` 执行；评审闭环（findings Resolution 回填、attended 模式人工确认）后，经 `hf-ship` 做 DoD 核验与关闭。缺陷工作项的 DoD 裁剪规则见 `hf-ship` 的 `references/definition-of-done.md`（第 1-3 项替换为「`fix.md` 的复现/根因/修复边界完整，复现测试先失败后通过」，其余各项不降低）。

**修复完成 ≠ 直接结束**：必须走 R3 评审与 `hf-ship` DoD。跳过这两步等于让缺陷修复不经独立检验与收口核验就进入主线。

> 收尾自检：[ ] `fix.md` 四问齐全；[ ] R3 评审闭环（findings Resolution 回填、attended 人工确认）；[ ] `hf-ship` DoD 核验后关闭；[ ] 未跳过 R3/ship。

## 紧急修复（hotfix）纪律

hotfix 可以**压缩文档**（`fix.md` 可精简到必要项），但**不能跳过任何质量门槛**。不可跳过的红线：

| 环节 | hotfix 是否可跳 |
|---|---|
| 复现（或记录已排除假设，证明不是盲补丁） | **不可跳** |
| 根因分析（至少直接原因 + 根本原因，波及范围可压缩） | **不可跳** |
| 先失败的复现测试（RED→GREEN） | **不可跳** |
| R3 代码/测试评审（`hf-review`） | **不可跳** |
| `hf-ship` DoD 核验与关闭 | **不可跳** |
| `fix.md` 详细叙述、波及范围全量排查 | 可压缩 |

紧急不等于绕过。线上火急时把火先灭（复现测试 + 最小修复 + R3 快审 + DoD），然后把省下的文档细节补回 `fix.md`——但任何"线上急着上，跳过测试/评审/DoD"的请求都应指出后果并坚持红线。

> hotfix 自检：[ ] 压缩的只是文档量，五条红线未跳；[ ] 没有以"紧急"为由省略复现/根因/测试/评审/DoD。

## 风险信号

- 没有复现记录就出现了修复 diff
- 复现不了却上了"盲修补丁"，或用重试/延时/放宽阈值让问题"不再出现"而不解释机理
- 根因写的是「空指针访问」这种"在哪发生"（直接原因），没有"为什么会发生"
- 修复 diff 里混着重命名、格式化、顺手清理
- 修复后没有新增任何测试（"改完手动验了"），或加了测试但没先看它失败
- 同一个函数第二次因同类原因被修（第一次的根因分析是假的）
- 行为/契约变更伪装成"修复"，没走 `hf-specify`
- 修复完成直接结束，跳过 R3 评审与 `hf-ship` DoD
- hotfix 以"紧急"为由跳过复现/根因/测试/评审/DoD

## 合理化反驳

| 话术 | 现实 |
|---|---|
| 「报告人没给复现步骤，我先改最可能的那行试试」 | 没复现就改是猜。先要复现（或记录已排除假设交回报告人）；盲补丁改完"不报错"不等于修好，真因可能还在 |
| 「不再报错了，应该修好了」 | "不再报错"可能只是把触发路径挪走或没踩到。先失败的复现测试转绿 + 完整套件无回归才是"修好"的证据 |
| 「根因是空指针」 | 那是直接原因（在哪发生）。根因要回答"为什么会漏判空"——契约不清？边界没定义？测试缺口？ |
| 「改一行试试看，不行再改一行」 | 这是猜，不是归因。两次试不中就回证据：读日志、二分、加诊断，别继续盲改 |
| 「flaky，加个重试稳一点」 | 重试/延时/放宽阈值让症状消失却不解释机理 = 掩盖。先提高复现率定位真因 |
| 「修复时顺手把周边坏味道一起清理了」 | 修复 diff 混入重命名/清理，评审者无法判读修复改了什么，回归出问题无法定位。登记，不顺手修 |
| 「契约/错误码变了，但这是修复」 | 行为/契约变更是变更不是修复，走 `hf-specify`（modify 条目）。伪装成修复会绕过规格评审 |
| 「缺陷嘛，手动验一下就行，不用写测试」 | 手动验证不可复现、不可回归。先写复现测试（RED）才是修复的凭据 |
| 「线上急着上，跳过测试和评审」 | 紧急不等于绕过。hotfix 压缩文档，不跳复现/根因/复现测试/R3/DoD 任何一条 |
| 「修完就结束，R3 和 ship 走个形式」 | 缺陷修复必须经 R3 评审与 hf-ship DoD 收口。跳过等于让未检验的修复进主线 |

## 验证清单

收尾前逐项确认：

- [ ] 复现步骤与环境已记录，证据可回看；不可复现的未上盲补丁
- [ ] 因果链每步有证据；直接原因 / 根本原因 / 波及范围三层齐全；已排除的假设记录了排除证据
- [ ] 修复边界显式：改什么、不改什么（带去向）、怎么回退
- [ ] 行为/契约变更走了 `hf-specify`，没有伪装成"修复"
- [ ] 存在先失败后通过的复现测试（RED→GREEN），失败原因是目标缺陷；完整套件无回归
- [ ] 测试缺口根因下补齐了同类边界测试
- [ ] 同类风险已登记（工作项/issue）；「为什么测试没拦住」已回答
- [ ] R3 评审闭环（findings Resolution 回填、attended 人工确认），经 `hf-ship` DoD 核验后关闭
- [ ] hotfix 只压缩文档，未跳复现/根因/测试/评审/DoD
