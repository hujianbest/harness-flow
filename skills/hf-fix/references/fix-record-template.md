# fix.md 模板

使用说明：`hf-fix` 生成组件根下 `features/DTS<id>-<slug>/fix.md`（或团队 `AGENTS.md` 覆盖路径）的默认模板。按严重度伸缩：简单缺陷可精简，但「复现」「根因三层」「先失败的复现测试」不可省。hotfix 可压缩文档，不能省这五条红线（复现 / 根因 / 复现测试 / R3 评审 / DoD）。

```markdown
# DTS<id> <标题>

## 复现

- 现象 / 影响范围 / 严重度:
- 环境: <版本/分支、平台/板卡、配置>
- 最小复现步骤:
  | 步骤 | 操作/输入 | 预期 | 实际 |
  |---|---|---|---|
- 证据: <日志/core/trace 路径>
- 复现稳定性: stable / flaky(频率+已尝试手段) / 不可复现(已排除假设见下)

## 根因

- 因果链（每步带证据）:
  现象 → … → 直接原因
- 直接原因（哪行代码/哪个配置错了）:
- 根本原因（为什么会发生：契约不清/边界未定义/测试缺口/…）:
- 为什么测试没拦住:
- 波及范围（同类模式还在哪里）:
- 已排除的假设:
  | 假设 | 排除证据 |
  |---|---|

## 修复边界

- 最小安全修复范围:
- 显式不修（登记去向）:
- 回退策略:
- 分流判断: 直接修复 / 走 hf-specify（契约变更）/ 走 hf-design（设计缺陷）

## 修复记录

- 复现测试: <测试名，RED 记录（命令+失败输出+commit 锚点）>
- 修复 diff 摘要: <GREEN commit，点明改了哪几处>
- 回归确认: <完整套件结果>
- 同类风险登记: <工作项/issue 链接>

## 收尾（R3 + hf-ship）

- R3 评审记录: <reviews/ 路径，findings Resolution 状态>
- hf-ship DoD: <closeout 路径或核验结论；缺陷工作项第 1-3 项替换为 fix.md 完整+复现测试先失败后通过>
```
